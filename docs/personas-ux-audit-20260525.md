# 케어브릿지72 사용자 페르소나 및 UX 문구 검수

작성일: 2026-05-25

## 1. 1차 페르소나

시군 통합돌봄 담당자

- 업무 상황: 퇴원 후 72시간 내 확인이 필요한 사례를 빠르게 검토하고, 후보 자원과 가족 안내 내용을 정리해야 한다.
- 제약: 의료 판단을 직접 내리거나 특정 기관을 지정하면 안 되며, 개인정보 입력을 최소화해야 한다.
- 기대: 한 화면에서 위험 신호, 후보 정보, 안내문, 판단 기록을 확인하고 인계할 수 있어야 한다.
- 적합한 문구: 사례, 위험 신호, 후보 정보, 확인, 검토, 인계, 처리 이력
- 피해야 할 문구: SaaS, API, fallback, prompt, model, LLM, referral, KPI dashboard

## 2. 2차 페르소나

병원 사회사업실 또는 퇴원지원팀 담당자

- 업무 상황: 퇴원 전후 사회경제적 공백, 보호자 연락 가능성, 지역 연계 필요성을 정리한다.
- 기대: 병원 기준정보와 지역 후보 정보를 구분하고, 시군 담당자에게 넘길 인계 요약을 빠르게 만든다.
- 중요한 UX: 환자에게 기관을 직접 연결하는 것처럼 보이지 않아야 하며, 담당자 확인 전 안내문이 확정된 것처럼 보이면 안 된다.

## 3. 보조 페르소나

공모전 평가자 또는 도입 검토자

- 업무 상황: 실제 동작 여부, 차별성, 데이터 활용 범위, 운영 경계를 짧은 시간 안에 확인한다.
- 기대: `/readiness`, `/capture`, `/api/health`에서 도입 가능성과 구현 상태를 확인한다.
- 중요한 UX: 사용자 업무 화면에는 평가자용 사업화 정보가 과도하게 노출되지 않아야 한다.

## 4. 전수 검수 결과

| 영역 | 발견 사항 | 조치 |
| --- | --- | --- |
| 메인 화면 | `SaaS`, `가명 데이터`가 업무자에게 다소 제품/개발자 관점으로 보임 | `담당자용 퇴원 사례 검토 화면`, `시연용 사례`로 수정 |
| 입력 폼 | `Preset`이 영어 UI 용어로 남아 있음 | `사례 선택`으로 수정 |
| 위험 표시 | `High/Medium/Low`가 영어로 표시됨 | `높음/중간/낮음`으로 수정 |
| 작업 화면 | `Workspace`, `Claude API 토큰`, `LLM`이 담당자에게 낯섦 | `작업 화면`, `초안 생성 비용`, `AI 사용 범위`로 수정 |
| 초안 카드 | `model`, `prompt`, `Claude 생성`, `fallback` 계열 운영 정보가 직접 노출됨 | `AI 생성`, `기본 초안`, `운영 정보`, `문구 기준`으로 수정 |
| 후보/병원 기준정보 | `외부 API` 표현이 개발자 중심 | `공공데이터 응답`, `기본 후보 정보`로 수정 |
| 이력 패널 | `감사 로그`가 딱딱하고 감사 대응용으로 보임 | `처리 이력`으로 수정 |
| 도입 검토 화면 | `Commercial readiness`, `PoC`, `deterministic fallback`, `API`가 혼재 | `도입 준비도`, `시범 운영`, `기본 초안 대체 구조`, `운영 상태 확인`으로 수정 |

## 5. 최종 UX 원칙

- 1차 사용자는 공공 담당자이므로 첫 화면은 `사업 설명`보다 `사례 검토 시작`을 우선한다.
- 후보 정보는 추천이 아니라 검토 대상이라는 표현을 유지한다.
- AI는 판단 주체가 아니라 문서 초안 작성 보조로만 설명한다.
- 기술 세부정보는 도입 검토와 문서에 두고, 업무 화면에는 필요한 만큼만 낮은 위계로 표시한다.
- 개인정보, 기관 지정, 법률·의료 판단으로 오해될 수 있는 문구는 입력·출력 양쪽에서 차단한다.
